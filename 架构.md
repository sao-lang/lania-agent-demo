## 1. 总体定位

本项目是一个 **Harness-first 的通用 Agent 运行时**，不是传统的 RAG 问答应用。

```text
传统 RAG 应用： 用户 → RAG Query Engine → 检索 → 生成 → 回答

Lania Agent Runtime：
  用户 → TaskSpec → Orchestrator → LangGraph 工作流
    → ExecutionHarness.run_tool(rag_*/repo_*/database_*/...) → Facade → Domain Service
```

**核心变化有三点**：

1. **RAG 降级为能力**：RAG 只是众多 Tool 之一，与 repository / database / api_contract 平级，不再是产品主轴
2. **Harness 是顶层**：治理（policy / guardrail / budget / sandbox / hook）由 Harness 控制，Agent 只负责任务推进
3. **ReAct 是局部的**：只在单 step 内运行有界循环（max_turns=3），不接管系统治理

## 2. 五层架构总览

系统分为六层+大脑层，从外到内依次是：

```text
┌────────────────────────────────────────────────────────────┐
│                     Entry Layer                            │
│     API / Service / Worker 入口、请求适配为 TaskSpec          │
├────────────────────────────────────────────────────────────┤
│                     State Layer                             │
│     Memory / Context / Checkpoint / Artifact Lineage        │
├────────────────────────────────────────────────────────────┤
│                    Workflow Layer                           │
│     LangGraph 编排 / Orchestrator / Skill / Step Lifecycle   │
├────────────────────────────────────────────────────────────┤
│                 ═══ Brain Layer ═══                        │
│     IntentRecognizer / ModeRouter / AgentLoop              │
│     StepExecutor / SafetyEngine                            │
│     ← 大脑在 Harness 中：大脑决定怎么调，Harness 约束怎么执行 → │
├────────────────────────────────────────────────────────────┤
│                   Capability Layer                          │
│     Tool / SubAgent / Facade Adapter / Capability Routing   │
│     CatalogTool / CustomizationEngine                      │
├────────────────────────────────────────────────────────────┤
│                   Governance Layer                          │
│     Policy / Guardrail / EventBus Hook / Sandbox / Budget   │
│     SafetyEngine（可插拔安全策略 7 种）                       │
├────────────────────────────────────────────────────────────┤
│                     Infra Layer                             │
│     LLM Provider / Vector Store / SQLite / Sandbox Worker   │
└────────────────────────────────────────────────────────────┘
```

**核心关系**：

```text
请求 → Entry → TaskSpec → Orchestrator → LangGraph 节点
  → [Brain Layer] IntentRecognizer → ModeRouter → AgentLoop
      → StepExecutor（含 SafetyEngine 检查）
        → ExecutionHarness.run_tool()
                                                      ↓
                                                 EventBus ──→ HookRegistry → Trace/Memory/Checkpoint
                                                      (每次 step/tool 事件都广播)
```

**一句话主链**：

```text
TaskSpec → LangGraph → Brain → ExecutionHarness → Tool → Facade
```

这条链之外不应有第二条平行主链。LangGraph 是唯一的工作流引擎，Brain Layer 是感知与决策中心。RAG 域内请求（/query、/chat）走独立 RAG 管道（QueryWorkflowOrchestrator → LangGraph → RagFacade），Agent 平台请求（/agent、/tasks）通过 Brain Layer → ExecutionHarness.run_tool() 统一调用所有 Tool。

## 3. 各层详解

### 3.1 Entry Layer（入口层）

#### 职责

接收外部请求，统一映射为 `TaskSpec`。不执行业务逻辑，只做适配。

#### 文件位置

| 文件 | 职责 |
|---|---|
| `app/api/v1/endpoints/query.py` | HTTP/SSE 接口：query / chat / stream_query / stream_chat |
| `app/api/v1/endpoints/tasks.py` | HTTP 接口：task CRUD、replay、resume |
| `app/services/query_service.py` | query/chat 请求入口 |
| `app/services/task_service.py` | task 请求入口 |
| `app/workflows/query_task_adapter.py` | QueryRequest/ChatRequest → TaskSpec |
| `app/workflows/tasks/document_analysis_task_adapter.py` | TaskRequest → TaskSpec |

#### 核心数据结构

```python
class TaskSpec(BaseModel):
    task_id: str                          # 唯一标识
    task_type: str                        # 'grounded_query' | 'session_chat' | 'document_analysis'
    objective: str                        # 任务目标描述
    input_payload: dict                   # 输入数据
    run_budget: RunBudget                 # 预算：max_steps, max_tool_calls, ...
    steps: list[StepSpec]                # 步骤声明
    success_criteria: list               # 成功标准
    created_at: datetime
```

#### 关键原则

- query / chat / task 三种请求都先映射为 `TaskSpec`
- Adapter 只做投影，不执行业务逻辑
- 外部不直接调 Agent 实例
- `query_stream` 和 `query` 的差异只在输出方式，不进入 TaskSpec

#### ContextBundle（Step 间传递的核心数据结构）

```python
class ContextBundle(BaseModel):
    step_id: str                     # 当前步骤 ID
    objective: str                   # 当前步骤目标
    
    # 数据切片
    state_slice: dict                # 任务状态切片
    evidence_slice: list[dict]       # 证据切片
    artifact_slice: dict | None      # 产物切片
    memory_slice: dict               # 记忆切片
    tool_options: list[str]          # 可用工具列表
    token_budget: int                # Token 预算
    
    # 可靠性元数据（记录"为什么给 step 这些数据"）
    source_summary: dict             # 各 slice 的来源，如 {"evidence_slice": "rag_retrieve_evidence/task123"}
    reliability_summary: dict        # 各 slice 的可靠性评估
    dropped_context_notes: list[str] # 被裁掉的上下文及原因
    context_version: int             # ContextBundle 版本
    built_at: str                    # 构建时间戳
```

#### MemoryRecord（记忆的通用单元）

```python
class MemoryRecord(BaseModel):
    memory_id: str
    scope: Literal['working', 'session', 'run', 'semantic', 'profile']
    namespace: dict                  # 隔离维度：tenant/user/task/run
    kind: Literal['observation', 'evidence', 'analysis', 'reflection', 'artifact', 'preference', 'error']
    trust_level: Literal['unverified', 'provisional', 'verified', 'final']
    source: Literal['tool', 'subagent', 'reflection', 'system', 'user']
    summary: str
    payload: dict
    degraded: bool
    stale: bool
    conflict_refs: list[str]
    checkpoint_ref: str | None
    related_task_run_id: str | None
    related_step_id: str | None
    created_at: datetime
```

#### 记忆分层模型

| scope 值 | 说明 | 生命周期 |
|---|---|---|
| `working` | 当前 step 临时工作集 | 当前步骤结束后清空或摘要下沉 |
| `session` | 同一 session 内的连续交互 | 默认不跨 session |
| `run` | 单次 TaskRun 的过程记忆 | 由 checkpoint 管理 |
| `semantic` | 跨 run 复用的稳定知识 | 延迟提交，严格门控 |
| `profile` | 用户/租户画像 | 长期，不允许单次任务改写 |

#### 工作流程

```text
Step 开始
  → ContextHarness.build_context(workflow_state, step_id)
  → 内部：Gather → Filter → Rank → Compress → Budget → Package
  → 产出：ContextBundle（含 source_summary + dropped_context_notes）
  → Step 消费 ContextBundle
  → Step 执行期间：ExecutionHarness.run_tool()
  → Step 完成后：ToolExecutionResult → TaskMemory.append_memory_record()
```

#### 关键原则

- `memory` 是仓库，`context` 是当前工作台
- 不是所有 memory 都进入 context
- 每轮 ReAct 都重建 context，不继承上轮完整工作集
- `append_memory_record()` 直接写 MemoryRecord，绕过旧的 TaskMemoryEntry 中间层

#### 架构设计：RAG 的双重身份

RAG 在系统中以两种身份运行：

```
RAG 作为独立应用（/query、/chat 端点）:
  QueryService → QueryWorkflowOrchestrator → LangGraph(Query Graph) → RagFacade（直接调用）
  ★ 不经过 ExecutionHarness，RAG 管道内部直接使用 RagFacade

RAG 作为注册 Tool（/agent、/tasks 端点）:
  AgentService/TaskService → Orchestrator → LangGraph 节点 → ExecutionHarness.run_tool("rag_*")
  ★ 所有 Tool 通过 ToolRegistry 统一调用，Harness 不知道具体 Capability 是什么
```

#### LangGraph 图执行模型

工作流由 **Orchestrator 组装 LangGraph StateGraph**，然后 `graph.invoke(state)` 执行。每个节点是独立的处理函数，条件路由由图的 conditional edges 决定。

**Query 工作流的 LangGraph 图结构：**

实际代码在 [app/workflows/query_graph.py](file:///e:/vsc-workspace/lania-zip/lania-agent-demo/app/workflows/query_graph.py) 的 `build_query_graph()` 中，核心设计是 **`dispatch_query_step` 路由器**：

```
__entry__ → check_guardrails
              ├─ blocked → blocked_response → finalize
              └─ dispatch → dispatch_query_step ← 核心路由器
                              ├─ load_session_context → 回到 dispatch
                              ├─ rewrite_query        → 回到 dispatch
                              ├─ expand_queries       → 回到 dispatch
                              ├─ lookup_cache
                              │    ├─ hit → cache_hit_response → finalize
                              │    └─ miss → 回到 dispatch
                              ├─ retrieve_evidence    → 回到 dispatch
                              ├─ compress_context     → 回到 dispatch
                              ├─ grounded_answer      → 回到 dispatch
                              ├─ self_reflect
                              │    ├─ retry → retry_retrieve → retrieve_evidence
                              │    ├─ rewrite → rewrite_answer → dispatch
                              │    └─ pass → 回到 dispatch
                              ├─ persist_session → finalize
                              └─ finalize → END
```

`dispatch_query_step` 类似 CPU 的取指-执行循环——每个 Step 节点执行完后都回到它，由它决定下一个要执行的 step。

**Task 工作流（以 DocumentAnalysis 为例）：**

```
plan → collect_document_context → retrieve_evidence → analyze
  → draft_report → review_report → revise_report → finalize_report
```

#### LangGraph 节点内部执行流程

每个 LangGraph 节点内部通过 `ExecutionHarness.run_tool()` 调用工具，治理链自动生效：

```text
LangGraph 节点函数（如 retrieve_evidence）
  │
  ├─ ContextHarness.build_context(workflow_state, step_id)  ← 构建上下文
  │
  ├─ ExecutionHarness.run_tool('rag_retrieve_evidence', payload, ws, ctx_bundle)
  │    │
  │    ├─ EventBus.before_tool()          ← 发射 before_tool 事件
  │    ├─ GuardrailEngine.validate_tool_call()  ← 护栏检查
  │    ├─ PolicyEngine.check_tool()             ← 权限检查
  │    ├─ SandboxEngine.assess()                ← 沙盒决策
  │    ├─ ToolExecutor.execute()                ← 执行（含 timeout/retry/circuit_breaker）
  │    ├─ EventBus.after_tool()           ← 发射 after_tool 事件
  │    └─ ExecutionHooks.record_execution()     ← Trace + Memory 写入
  │
  └─ 返回更新后的 state
```

#### Step 生命周期管理

`app/workflows/step_lifecycle.py` 提供共享的 Step 生命周期 helper，query 和 task 两侧共用：

- `step_started` / `step_completed` / `step_failed` 事件
- `checkpoint` 创建（在关键步骤后自动保存）
- `RunEvent` 记录（进入 TaskRun 数据面）

#### 关键原则

- **LangGraph 是唯一的工作流引擎**——不在其上叠加 Recipe/Stage/HarnessKernel 抽象
- **Orchestrator 是组装者**——负责创建 LangGraph 图、注入依赖、驱动执行
- **RAG 是独立应用 + 也是 Tool**——`/query` `/chat` 走自己的 RAG 管道；同时 RAG 能力注册为 Tool 供 Agent 平台调用
- **ExecutionHarness 只依赖 ToolRegistry**——不 import 任何具体 Capability
- 新增任务类型时写 Skill + LangGraph 图，不改 Orchestrator 骨架
- 节点不直接调 `trace.record()` 或写内存——这些通过 EventBus 由 hook 处理

### 3.4 Brain Layer（大脑层）— 新增

> **核心命题**：大脑在 Harness 中。大脑决定怎么调用，Harness 约束怎么执行。

#### 职责

感知输入、识别意图、路由模式、驱动 Agent Loop，并在每一步执行时通过 SafetyEngine 做安全检查。大脑层 = 感知层（IntentRecognizer）+ 决策层（AgentLoop）+ 安全层（SafetyEngine）。

#### 架构分层

```text
┌── harness/brain/ ──────────────────────────────────────────┐
│                                                             │
│  ┌─ 感知层（Perception）— 一次调用，设定上下文 ───────────┐ │
│  │  IntentRecognizer → 这是什么问题？需要什么知识来源？    │ │
│  │                      复杂度多高？风险有多大？建议什么模式？│ │
│  │  输出: IntentDecision（一次性，会话开始时）              │ │
│  └─────────────────────────────────────────────────────────┘ │
│                            │                                  │
│                            ▼                                  │
│  ┌─ 决策层（Decision）— 多轮循环，在上下文中持续决策 ────┐ │
│  │  ModeRouter → 根据 IntentDecision 决定最终执行模式     │ │
│  │  AgentLoop  → 在确定模式下，逐轮决策：                 │ │
│  │                调哪个工具？参数是什么？够了没有？       │ │
│  │  输出: 多轮工具调用 + 最终回答                        │ │
│  └─────────────────────────────────────────────────────────┘ │
│                                                             │
│  全部 LLM 驱动，不是关键词驱动                               │
└─────────────────────────────────────────────────────────────┘
```

#### 文件位置

| 文件 | 职责 |
|---|---|
| `app/harness/brain/intent_recognizer.py` | 统一意图识别：双层策略（QuickHeuristicClassifier <1ms + LLMIntentClassifier ~200ms） |
| `app/harness/brain/mode_router.py` | 模式路由：IntentDecision → SuggestedMode（chat/plan/autopilot） |
| `app/harness/brain/agent_loop.py` | LLM 驱动的工具调用循环：LLM 决定 → StepExecutor 执行 → 结果回传 |
| `app/harness/brain/step_executor.py` | 步骤执行器：SafetyEngine 检查 → 用户确认 → 服务端/客户端路由 |
| `app/harness/brain/models.py` | 大脑层数据模型：IntentDecision / RouteResult / SafetyDecision / ToolCall / ConsentRecord |

#### 执行流程

```text
1. IntentRecognizer.recognize(user_input, context)
   → IntentDecision（knowledge_sources, complexity, risk_level, suggested_mode）

2. ModeRouter.route(intent_decision, route_context)
   → SuggestedMode（chat / plan / autopilot）

3. AgentLoop.run(mode, user_input, context)
   └─ 循环（LLM 决定 → StepExecutor 执行 → 评估结果）:
      ├─ StepExecutor.execute(tool_call, context)
      │   ├─ SafetyEngine.pre_execution_check(tool_call)  ← 执行前安全检查
      │   ├─ 执行工具（服务端沙箱 / 客户端下发）
      │   ├─ SafetyEngine.post_output_check(result)       ← 输出安全检查
      │   └─ ConsentStore（需要时记录"记住此选择"）
      └─ 直到完成 / 需要用户输入 / 达到上限
```

#### SafetyEngine（安全引擎）

5 层可插拔安全策略体系，部署者可编写自定义策略：

| 策略 | 文件 | 检查内容 |
|---|---|---|
| 数据销毁 | `app/harness/safety/policies/data_destruction.py` | 防止破坏性操作 |
| 数据外泄 | `app/harness/safety/policies/data_exfiltration.py` | 防止敏感数据泄露 |
| 权限提升 | `app/harness/safety/policies/privilege_escalation.py` | 防止越权行为 |
| 远程代码执行 | `app/harness/safety/policies/remote_code_execution.py` | 防止恶意代码执行 |
| 会话风险 | `app/harness/safety/policies/session_risk.py` | 会话级风险检测 |
| 系统篡改 | `app/harness/safety/policies/system_tampering.py` | 防止系统配置篡改 |
| 工具输出内容 | `app/harness/safety/policies/tool_output_content.py` | 工具输出内容安全 |

```python
class SafetyPolicy(ABC):
    """可插拔安全策略插件"""
    name: str
    def check(self, context: SafetyContext) -> SafetyDecision: ...
```

通过 `config/safety.yaml` 配置启用/禁用、严重级别、白名单/黑名单。

#### 关键原则

- **大脑在 Harness 中**：大脑决定怎么调，Harness（Guardrail/Policy/Sandbox）约束怎么执行
- **感知层只执行一次**：IntentRecognizer 在会话开始时输出 IntentDecision，不逐轮重识别
- **决策层持续交互**：AgentLoop 多轮循环，在上下文中持续决策
- **安全可插拔**：SafetyPolicy 插件化，部署者按需启用/编写策略

#### ToolRegistry 职责

```python
class ToolRegistry:
    def register(self, tool: AgentTool) -> None: ...      # 注册工具
    def get(self, name: str) -> AgentTool: ...             # 按名称取工具
    def describe(self, name: str) -> ToolSchema: ...       # 返回 schema
    def list_descriptions(self) -> list[ToolSchema]: ...   # 返回所有 schema
    def run(self, name, payload, context, *, sandbox_runner=None): ...   # 执行
```

`ToolRegistry.run()` 内部流程：

```text
1. 从 _tools 中取出工具实例
2. 校验 payload 是否符合 input_model
3. 如果 sandbox_runner 存在，走沙盒执行；否则直接调 tool.run()
4. 校验 result 是否符合 output_model
5. 记录 tool call 到 TaskMemory（含 duration_ms / input_preview / output_summary / error）
6. 记录 trace 事件
```

#### ToolExecutor（治理执行器）

```python
class ToolExecutor:
    def execute(self, tool_name, payload, workflow_state, tool_call_id, runtime_policy, sandbox_mode):
        # 带 timeout 执行
        # 带 retry（根据 policy.max_attempts）
        # 带熔断（连续失败超过 threshold 后自动拒绝）
        # 返回 ToolExecutionOutcome(result, retry_count, attempts)
    
    def mark_circuit_failure(self, tool_name, error, runtime_policy): ...
    def is_circuit_open(self, tool_name) -> bool: ...
```

#### ExecutionHarness（治理编排）

`ExecutionHarness.run_tool()` 是工具调用的**完整治理链**入口：

```text
run_tool(name, payload, workflow_state, context_bundle)
  │
  ├─ 1. EventBus.emit_before_tool()          # 发射 before_tool 事件
  │
  ├─ 2. GuardrailEngine.validate_tool_call()  # 护栏检查
  │    └─ 不通过 → 抛 ToolExecutionError
  │
  ├─ 3. PolicyEngine.check_tool()             # 权限检查
  │    └─ 不通过 → 抛 permission_error
  │
  ├─ 4. SandboxEngine.assess()                # 沙盒决策
  │    └─ risk_level=high + step 不在白名单 → 拒绝
  │    └─ 返回 sandbox_mode（inline / process_isolated）
  │
  ├─ 5. ToolExecutor.execute()                # 执行（含 timeout/retry/circuit_breaker）
  │    ├─ 成功 → 返回 result
  │    └─ 失败 → FallbackHandler.apply() 或 抛异常
  │
  ├─ 6. EventBus.emit_after_tool()            # 发射 after_tool 事件
  │
  └─ 7. ExecutionHooks.record_execution()     # Trace 记录 + RuntimeSummary 写回 TaskMemory
```

#### ToolContext（工具运行时依赖）

```python
@dataclass
class ToolContext:
    state: InMemoryState         # 状态存储
    retrieval: Any               # 检索服务
    trace: TraceRecorder         # 链路追踪
    task_memory: TaskMemory      # 任务记忆
    settings: Settings           # 全局配置
    llm: Any | None = None
    vector_store: Any | None = None
    deps: dict[str, Any] | None = None     # 动态依赖注入（替代硬编码 Capability 字段）
    services: dict[str, Any] | None = None # 外部服务（weather/finance/...）
    task_id: str | None = None
    step_name: str | None = None
    tool_call_id: str | None = None
    run_budget: Any | None = None
    model_router: ModelRouter | None = None

    def __getattr__(self, name: str) -> Any:
        """委托到 deps dict，工具可通过 context.rag / context.knowledge 等访问。"""
        if name == 'deps' or name.startswith('__'):
            raise AttributeError(name)
        deps = self.deps or {}
        if name in deps:
            return deps[name]
        raise AttributeError(name)
```

`deps` 字典是**所有领域依赖的统一注入点**。工具通过 `context.rag`、`context.knowledge`、`context.repository` 等访问对应 Capability（通过 `__getattr__` 委托到 `deps` dict），取不到时自动抛 `AttributeError`。

`services` 字典是**外部数据服务**的统一注入点。工具通过 `context.services.get('weather')` 或
`context.services.get('finance')` 获取对应 Capability，取不到时自动回退到自建实例。

#### 事件发射→路由→消费

```text
LangGraph 节点 / ExecutionHarness 发射事件
  → event_bus.before_tool(ws, tool_name='rag_retrieve_evidence')
    → HookRegistry.emit(EventPayload(event='before_tool', ...))

      ├→ MemoryHook.handle(event)        # 特定事件监听 → TaskMemory.append_memory_record(...)
      └→ (其他注册的 hook)
```

#### HookRegistry 特性

- 支持通配符 `'all'` 注册：一个 hook 监听所有事件（如 TraceHook）
- 支持按事件类型注册：一个 hook 只监听特定事件
- 执行顺序：通配符 handler 先跑，特定事件 handler 后跑

```text
  ├─ react_runtime: BoundedLocalReActRuntime
  ├─ reflection_harness: ReflectionHarness
  ├─ recovery_manager: RecoveryManager
  ├─ model_router: ModelRouter
  └─ rag_facade: RagFacade
```

### 5.5 模块间数据传递边界

| 边界 | 数据类型 | 协议/方式 |
|---|---|---|
| Entry → Workflow | `TaskSpec + StepSpec` | Python 对象 |
| Workflow → State | `ContextBundle (from ContextHarness)` | Python 对象 |
| Workflow → Capability | `tool_name + payload dict + RuntimeContext` | ToolExecutor.execute() |
| Capability → Governance | `ToolExecutionResult / ExecutionRuntimeSummary` | EventBus 事件 |
| Governance → State | `MemoryRecord` | TaskMemory.append_memory_record() |
| Governance → Trace | `event payload dict` | TraceRecorder.record() |
| LangGraph 节点 → EventBus | `HookEvent + workflow_state + payload` | EventBus.before_tool/after_tool |
| EventBus → Hook | `EventPayload` | HookRegistry.emit() → RuntimeHook.handle() |
| Container → ToolContext | `external_services` dict | `container.py` → `Orchestrator` → `ExecutionHarness` → `ToolExecutor._tool_context()` → `ToolContext.services` |

## 6. 关键业务场景分析

### 6.1 新增一个 Task 类型需要改什么

```text
1. 写一个 Skill 实现（定义任务初始状态和步骤）
2. 写一个 LangGraph 图定义（定义节点和路由）
3. 注册到 TaskSkillRegistry
```

**不需要改**：ToolExecutor、PolicyEngine、GuardrailEngine、EventBus、ExecutionHarness

### 6.2 新增一个 Tool 需要改什么

```text
1. 写 Tool 子类（实现 run() 方法，定义 input_model / output_model）
2. 注册到 ToolRegistry（一行：registry.register(MyTool())）
```

如果 Tool 需要调用外部 API（天气/金融/新闻等）：

```text
1. 在 app/capabilities/<domain>/ 下写 Capability 实现（API 调用逻辑）
2. 在 app/agents/tools/ 下写 Tool 子类（通过 context.services.get('name') 获取 Capability）
3. 在 container.py 中：初始化 Capability + 加入 external_services 字典 + 注册 Tool
4. （可选）在 app/core/config.py 中添加 API Key 环境变量
```

如果 Tool 需要通过定制化原语（CatalogTool）暴露给 LLM：

```text
1. 将扩展内容放到 .agents/skills/<name>/ 或 .agents/instructions/ 下
2. CustomizationEngine 自动扫描加载，ExtensionCatalog 自动注册到 LLM 可见清单
3. LLM 通过 load_extension / load_rule 工具按需加载
```

**不需要改**：ToolExecutor、PolicyEngine、GuardrailEngine、ExecutionHarness

### 6.4 新增一个安全策略需要改什么

```text
1. 实现 SafetyPolicy 子类（class MyPolicy: name + check(context) → SafetyDecision）
2. 注册到 SafetyEngine（一行：engine.register(MyPolicy())）
3. 在 config/safety.yaml 中配置策略行为
```

**不需要改**：StepExecutor、AgentLoop、ExecutionHarness

### 6.5 新增一个治理策略需要改什么

```text
1. 实现 RuntimeHook 协议（class MyHook: name + handle(event)）
2. 注册到 HookRegistry（一行：bus.register(MyHook(), event=HookEvent.BEFORE_TOOL)）
```

**不需要改**：ToolExecutor、ExecutionHarness（除非要新的治理链阶段）

### 6.6 新增一个 Infra Provider（如新 LLM / 新 Vector Store）需要改什么

```text
1. 在 app/rag 或对应域下实现 Provider
2. 在 container.py 中装配并注入对应的 Facade
```

## 7. 核心设计原则

### 原则 1：每类能力只有一个主管理层

| 能力 | 管理归属 | 注册方式 |
|---|---|---|
| session / memory / context | State Layer | TaskMemory / ContextHarness |
| skill / workflow | Workflow Layer | TaskSkillRegistry / LangGraph 图 |
| tool / command / sub-agent | Capability Layer | ToolRegistry / SubAgentRegistry |
| external service / domain API | Capability Layer | `external_services` dict → ToolContext |
| permission / policy / hook / audit | Governance Layer | PolicyEngine / HookRegistry |
| tech-stack | Infra Layer | Settings / Container |

### 原则 2：扩展靠继承，执行靠组合

**通过继承扩展**：

```python
class BaseTool        # 新增能力时继承
class BaseSubAgent    # 新增受控子代理时继承
```

**通过组合执行**：

```python
# Orchestrator 组装 LangGraph 图，由 graph.invoke() 驱动执行
graph = build_query_graph(tool_registry, execution_harness, ...)
final_state = graph.invoke(initial_state)
EventBus + HookRegistry                 # 治理组合由事件路由
```

### 原则 3：Harness > Agent

- Budget / Permission / Guardrail / Sandbox 由 Harness 控制
- Agent 只负责任务推进
- ReAct 只用于 step 内局部动作选择

### 原则 4：领域能力不反穿 Harness

```text
正确路径：LangGraph 节点 → ExecutionHarness.run_tool() → Tool → Facade → Domain Service
错误路径：LangGraph 节点 → 直接调 KnowledgeCapability / RagQueryEngine
```

- Workflow 不直接调 `KnowledgeCapability`
- RAG 通过 `rag_*` tool + `RagFacade` 接入
- 新增 domain 能力时：Facade → ToolAdapter → Tool → ToolRegistry

### 原则 5：治理不散落在节点中

```text
好的方式：节点只发射事件 → EventBus → HookRegistry → TraceHook/MemoryHook
不好的方式：节点内部自己调 trace.record() + 自己写 TaskMemory + 自己判断权限
```

## 10. 一句话总结

系统以 `TaskSpec → LangGraph → ExecutionHarness → Tool → Facade` 为主线，`EventBus` 为治理总线，通过五层架构让 RAG 降级为能力、ReAct 受约束、治理收口为平台层——最终做到**每类能力只有一个主管理层，新增扩展只需注册不需改骨架**。
