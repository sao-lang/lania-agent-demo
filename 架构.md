## 1. 总体定位

本项目是一个 **Harness-first 的通用 Agent 运行时**，不是传统的 RAG 问答应用。

```text
传统 RAG 应用： 用户 → RAG Query Engine → 检索 → 生成 → 回答

Lania Agent Runtime：
  用户 → TaskSpec → HarnessKernel → Recipe/Stage
    → Tool(rag_*/repo_*/database_*/...) → Facade → Domain Service
```

**核心变化有三点**：

1. **RAG 降级为能力**：RAG 只是众多 Tool 之一，与 repository / database / api_contract 平级，不再是产品主轴
2. **Harness 是顶层**：治理（policy / guardrail / budget / sandbox / hook）由 Harness 控制，Agent 只负责任务推进
3. **ReAct 是局部的**：只在单 step 内运行有界循环（max_turns=3），不接管系统治理

## 2. 五层架构总览

系统分为五层，从外到内依次是：

```text
┌────────────────────────────────────────────────────────────┐
│                     Entry Layer                            │
│     API / Service / Worker 入口、请求适配为 TaskSpec          │
├────────────────────────────────────────────────────────────┤
│                     State Layer                             │
│     Memory / Context / Checkpoint / Artifact Lineage        │
├────────────────────────────────────────────────────────────┤
│                    Workflow Layer                           │
│     Recipe / Stage / Step Lifecycle / LangGraph 编排        │
├────────────────────────────────────────────────────────────┤
│                   Capability Layer                          │
│     Tool / SubAgent / Facade Adapter / Capability Routing   │
├────────────────────────────────────────────────────────────┤
│                   Governance Layer                          │
│     Policy / Guardrail / EventBus Hook / Sandbox / Budget   │
├────────────────────────────────────────────────────────────┤
│                     Infra Layer                             │
│     LLM Provider / Vector Store / SQLite / Sandbox Worker   │
└────────────────────────────────────────────────────────────┘
```

**核心关系**：

```text
请求 → Entry → TaskSpec → HarnessKernel → Recipe → Stage → ToolExecutor
                                        ↓                          ↓
                                   EventBus ──→ HookRegistry → Trace/Memory/Checkpoint
                                        (每次 stage/tool 事件都广播)
```

**一句话主链**：

```text
TaskSpec → Recipe → Stage → Executor → Tool → Facade
```

这条链之外不应有第二条平行主链。

## 3. 各层详解

### 3.1 Entry Layer（入口层）

#### 职责

接收外部请求，统一映射为 `TaskSpec`。不执行业务逻辑，只做适配。

#### 文件位置

| 文件 | 职责 |
|---|---|
| `app/api/v1/endpoints/query.py` | HTTP/SSE 接口：query / chat / stream_query / stream_chat |
| `app/api/v1/endpoints/tasks.py` | HTTP 接口：task CRUD、replay、resume |
| `app/services/query_service.py` | query/chat 请求入口 |
| `app/services/task_service.py` | task 请求入口 |
| `app/workflows/query_task_adapter.py` | QueryRequest/ChatRequest → TaskSpec |
| `app/workflows/tasks/document_analysis_task_adapter.py` | TaskRequest → TaskSpec |

#### 核心数据结构

```python
class TaskSpec(BaseModel):
    task_id: str                          # 唯一标识
    task_type: str                        # 'grounded_query' | 'session_chat' | 'document_analysis'
    objective: str                        # 任务目标描述
    input_payload: dict                   # 输入数据
    run_budget: RunBudget                 # 预算：max_steps, max_tool_calls, ...
    steps: list[StepSpec]                # 步骤声明
```

#### 关键原则

- query / chat / task 三种请求都先映射为 `TaskSpec`
- Adapter 只做投影，不执行业务逻辑
- 外部不直接调 Agent 实例
- `query_stream` 和 `query` 的差异只在输出方式，不进入 TaskSpec

#### ContextBundle（Step 间传递的核心数据结构）

```python
class ContextBundle(BaseModel):
    step_id: str                     # 当前步骤 ID
    objective: str                   # 当前步骤目标
    
    # 数据切片
    state_slice: dict                # 任务状态切片
    evidence_slice: list[dict]       # 证据切片
    artifact_slice: dict | None      # 产物切片
    memory_slice: dict               # 记忆切片
    tool_options: list[str]          # 可用工具列表
    token_budget: int                # Token 预算
    
    # 可靠性元数据（记录"为什么给 step 这些数据"）
    source_summary: dict             # 各 slice 的来源，如 {"evidence_slice": "rag_retrieve_evidence/task123"}
    reliability_summary: dict        # 各 slice 的可靠性评估
    dropped_context_notes: list[str] # 被裁掉的上下文及原因
    context_version: int             # ContextBundle 版本
    built_at: str                    # 构建时间戳
```

#### MemoryRecord（记忆的通用单元）

```python
class MemoryRecord(BaseModel):
    memory_id: str
    scope: Literal['working', 'session', 'run', 'semantic', 'profile']
    namespace: dict                  # 隔离维度：tenant/user/task/run
    kind: Literal['observation', 'evidence', 'analysis', 'reflection', 'artifact', 'preference', 'error']
    trust_level: Literal['unverified', 'provisional', 'verified', 'final']
    source: Literal['tool', 'subagent', 'reflection', 'system', 'user']
    summary: str
    payload: dict
    degraded: bool
    stale: bool
    conflict_refs: list[str]
    checkpoint_ref: str | None
    related_task_run_id: str | None
    related_step_id: str | None
    created_at: datetime
```

#### 记忆分层模型

| scope 值 | 说明 | 生命周期 |
|---|---|---|
| `working` | 当前 step 临时工作集 | 当前步骤结束后清空或摘要下沉 |
| `session` | 同一 session 内的连续交互 | 默认不跨 session |
| `run` | 单次 TaskRun 的过程记忆 | 由 checkpoint 管理 |
| `semantic` | 跨 run 复用的稳定知识 | 延迟提交，严格门控 |
| `profile` | 用户/租户画像 | 长期，不允许单次任务改写 |

#### 工作流程

```text
Step 开始
  → ContextHarness.build_context(workflow_state, step_id)
  → 内部：Gather → Filter → Rank → Compress → Budget → Package
  → 产出：ContextBundle（含 source_summary + dropped_context_notes）
  → Step 消费 ContextBundle
  → Step 执行期间：ExecutionHarness.run_tool()
  → Step 完成后：ToolExecutionResult → TaskMemory.append_memory_record()
```

#### 关键原则

- `memory` 是仓库，`context` 是当前工作台
- 不是所有 memory 都进入 context
- 每轮 ReAct 都重建 context，不继承上轮完整工作集
- `append_memory_record()` 直接写 MemoryRecord，绕过旧的 TaskMemoryEntry 中间层

```python
    def __init__(self):
        super().__init__(stages=[
            GuardrailStage(),          # 安全检查
            RewriteStage(),            # 查询改写
            RetrieveEvidenceStage(),   # 证据检索
            GroundedAnswerStage(),     # 生成回答

class RetrieveEvidenceStage(BaseStage):
```

#### HarnessKernel 执行流程

```python
kernel = HarnessKernel(event_bus=bus)
result = kernel.run(recipe, state, ctx, workflow_state=ws)

# 内部执行顺序：
# 1. event_bus.run_started(ws, recipe_name=...)
# 2. for stage in recipe.stages():
# 3.    event_bus.before_stage(ws, stage_name)
# 4.    try:
# 5.        new_state = stage.run(state, ctx)
# 6.    except Exception as exc:
# 7.        event_bus.stage_failed(ws, stage_name, error=...)
# 8.        stage_errors[stage_name] = str(exc)
# 9.        return HarnessResult(completed=False, failed_stage=stage_name)
# 10.   state = new_state
# 11.   completed_stage_ids.append(stage_name)
# 12.   event_bus.after_stage(ws, stage_name)
# 13. event_bus.run_completed(ws, completed_stages=...)
```

#### Kernel 与 LangGraph 的关系

```text
HarnessKernel 是"业务编排层"——决定任务推进顺序
       ↓
Orchestrator 内部用 Recipe 表达流程，实际执行调 LangGraph compiled graph
       ↓
LangGraph 是"图执行器"——执行节点和路由
       ↓
graph 内部每个 node 对应一个或多个 Stage 语义
```

#### RecipeRegistry

```python
class RecipeRegistry:
    def register(self, recipe: BaseRecipe) -> None: ...
    def get(self, name: str) -> BaseRecipe: ...
    def get_by_task_type(self, task_type: str) -> BaseRecipe | None: ...

    def list_meta(self) -> list[RecipeMeta]: ...
```

同时按 `recipe.name` 和 `recipe.task_type` 建立索引。

#### 关键原则

- Recipe/Stage 是业务的声明式表达，不是执行细节
- `Node` 退回 graph 实现细节，不是对外建模对象
- 新增任务类型时只需写 Recipe + Stage，不改 kernel
- Stage 不直接调 `trace.record()` 或写内存——这些通过 EventBus 由 hook 处理

#### ToolRegistry 职责

```python
class ToolRegistry:
    def register(self, tool: AgentTool) -> None: ...      # 注册工具
    def get(self, name: str) -> AgentTool: ...             # 按名称取工具
    def describe(self, name: str) -> ToolSchema: ...       # 返回 schema
    def list_descriptions(self) -> list[ToolSchema]: ...   # 返回所有 schema
    def run(self, name, payload, context, *, sandbox_runner=None): ...   # 执行
```

`ToolRegistry.run()` 内部流程：

```text
1. 从 _tools 中取出工具实例
2. 校验 payload 是否符合 input_model
3. 如果 sandbox_runner 存在，走沙盒执行；否则直接调 tool.run()
4. 校验 result 是否符合 output_model
5. 记录 tool call 到 TaskMemory（含 duration_ms / input_preview / output_summary / error）
6. 记录 trace 事件
```

#### ExecutionHarness（治理编排）

`ExecutionHarness.run_tool()` 是工具调用的**完整治理链**入口：

```text
run_tool(name, payload, workflow_state, context_bundle)
  │
  ├─ 1. EventBus.emit_before_tool()          # 发射 before_tool 事件
  │
  ├─ 2. GuardrailEngine.validate_tool_call()  # 护栏检查
  │    └─ 不通过 → 抛 ToolExecutionError
  │
  ├─ 3. PolicyEngine.check_tool()             # 权限检查
  │    └─ 不通过 → 抛 permission_error
  │
  ├─ 4. SandboxEngine.assess()                # 沙盒决策
  │    └─ risk_level=high + step 不在白名单 → 拒绝
  │    └─ 返回 sandbox_mode（inline / process_isolated）
  │
  ├─ 5. ToolExecutor.execute()                # 执行（含 timeout/retry/circuit_breaker）
  │    ├─ 成功 → 返回 result
  │    └─ 失败 → FallbackHandler.apply() 或 抛异常
  │
  ├─ 6. EventBus.emit_after_tool()            # 发射 after_tool 事件
  │
  └─ 7. ExecutionHooks.record_execution()     # Trace 记录 + RuntimeSummary 写回 TaskMemory
```

#### EventBus 事件模型

```python
class HookEvent(str, Enum):
    # Stage 生命周期 (3 种)
    BEFORE_STAGE = 'before_stage'
    AFTER_STAGE = 'after_stage'
    STAGE_FAILED = 'stage_failed'

    # Tool 生命周期 (3 种)
    BEFORE_TOOL = 'before_tool'
    AFTER_TOOL = 'after_tool'
    TOOL_FAILED = 'tool_failed'

    # ReAct 生命周期 (3 种)
    BEFORE_REACT_TURN = 'before_react_turn'
    AFTER_REACT_TURN = 'after_react_turn'
    REACT_EXCEEDED_MAX_TURNS = 'react_exceeded_max_turns'

    # Checkpoint 生命周期 (2 种)
    BEFORE_CHECKPOINT = 'before_checkpoint'
    AFTER_CHECKPOINT = 'after_checkpoint'

    # Request/Run 生命周期 (3 种)
    RUN_STARTED = 'run_started'
    RUN_COMPLETED = 'run_completed'
    RUN_FAILED = 'run_failed'

    # Recovery 生命周期 (3 种)
    RECOVERY_INITIATED = 'recovery_initiated'
    RECOVERY_COMPLETED = 'recovery_completed'
    RECOVERY_FAILED = 'recovery_failed'

    # Context 生命周期 (2 种)
    CONTEXT_BUILT = 'context_built'
    CONTEXT_TRIM = 'context_trim'
```

#### 事件发射→路由→消费

```text
HarnessKernel.run()
  → event_bus.before_stage(ws, stage_name='retrieve')
    → HookRegistry.emit(EventPayload(event='before_stage', ...))
      ├→ TraceHook.handle(event)         # 通配符监听 → trace.record('before_stage', ...)
      ├→ MemoryHook.handle(event)        # 特定事件监听 → TaskMemory.append_memory_record(...)
      └→ (其他注册的 hook)
```

#### HookRegistry 特性

- 支持通配符 `'all'` 注册：一个 hook 监听所有事件（如 TraceHook）
- 支持按事件类型注册：一个 hook 只监听特定事件
- 执行顺序：通配符 handler 先跑，特定事件 handler 后跑

#### Sandbox 体系（三层）

```text
1. ToolSandbox（已有的）
   工具级隔离：路径白名单、网络白名单、高风险工具进程隔离

2. ContextSandbox（新增）
   上下文级裁剪：控制 sub-agent/tool 能看到的 state/evidence/artifact/memory 范围
   → ContextAccessPolicy 定义 allowed_state_keys / blocked_state_keys / max_evidence_items / ...

3. CapabilitySandbox（新增）
   能力级约束：按 capability 类型预先设定默认限制
   → knowledge 只查知识库、filesystem 只读工作区、shell 禁止并发
```

ContextSandbox 示例：

```python
DEFAULT_TOOL_CONTEXT_POLICY = ContextAccessPolicy(
    blocked_state_keys=['task.request.api_key'],
    allow_artifact=False,
    max_memory_items=10,
    max_evidence_items=20,
    allow_session_data=False,
)

sandbox = ContextSandbox()
decision = sandbox.filter(
    bundle=context_bundle,
    policy_name='tool',
    extra_keys=['current_step'],
)
# decision.filtered_bundle 是裁剪后的 bundle
# decision.removed_notes = ['memory_slice: cut to 10 keys', 'artifact_slice: disabled by policy']
```

```python
class PromptVersionRegistry:
    def register_version(self, prompt_id, content, output_schema, change_log, *, activate=True): ...
    def get_active_version(self, prompt_id) -> PromptVersion: ...
    def rollback_to(self, prompt_id, version) -> PromptVersion: ...
```

## 6. 关键业务场景分析

### 6.1 新增一个 Task 类型需要改什么

```text
1. 写一个 Skill 子类（如果 task 需要特殊初始状态）
2. 写一个 Recipe + 若干 BaseStage 子类（声明执行顺序）
3. 注册到 RecipeRegistry / SkillRegistry
```

**不需要改**：HarnessKernel、ToolExecutor、PolicyEngine、GuardrailEngine、EventBus

### 6.2 新增一个 Tool 需要改什么

```text
1. 写 Tool 子类（实现 run() 方法，定义 input_model / output_model）
2. 注册到 ToolRegistry（一行：registry.register(MyTool())）
```

**不需要改**：HarnessKernel、Stage、ToolExecutor、PolicyEngine、GuardrailEngine

### 6.3 新增一个治理策略需要改什么

```text
1. 实现 RuntimeHook 协议（class MyHook: name + handle(event)）
2. 注册到 HookRegistry（一行：bus.register(MyHook(), event=HookEvent.BEFORE_TOOL)）
```

**不需要改**：HarnessKernel、Stage、ToolExecutor（除非要新的治理链阶段）

### 6.4 新增一个 Infra Provider（如新 LLM / 新 Vector Store）需要改什么

```text
1. 在 app/rag 或对应域下实现 Provider
2. 在 container.py 中装配并注入对应的 Facade
```

## 7. 核心设计原则

### 原则 1：每类能力只有一个主管理层

| 能力 | 管理归属 | 注册方式 |
|---|---|---|
| session / memory / context | State Layer | TaskMemory / ContextHarness |
| skill / recipe / stage | Workflow Layer | TaskSkillRegistry / RecipeRegistry |
| tool / command / sub-agent | Capability Layer | ToolRegistry / SubAgentRegistry |
| permission / policy / hook / audit | Governance Layer | PolicyEngine / HookRegistry |
| tech-stack | Infra Layer | Settings / Container |

### 原则 2：扩展靠继承，执行靠组合

**通过继承扩展**：

```python
class BaseRecipe      # 新增 task 类型时继承
class BaseStage       # 新增 step 类型时继承
class BaseTool        # 新增能力时继承
class BaseSubAgent    # 新增受控子代理时继承
```

**通过组合执行**：

```python
recipe = QueryRecipe(stages=[GuardrailStage(), RewriteStage(), ...])
kernel.run(recipe, state, ctx)          # 组合执行由 kernel 驱动
EventBus + HookRegistry                 # 治理组合由事件路由
```

### 原则 3：Harness > Agent

- Budget / Permission / Guardrail / Sandbox 由 Harness 控制
- Agent 只负责任务推进
- ReAct 只用于 step 内局部动作选择

### 原则 4：领域能力不反穿 Harness

```text
正确路径：Stage → Tool → Facade → Domain Service
错误路径：Stage → 直接调 KnowledgeCapability / RagQueryEngine
```

- Workflow 不直接调 `KnowledgeCapability`
- RAG 通过 `rag_*` tool + `RagFacade` 接入
- 新增 domain 能力时：Facade → ToolAdapter → Tool → Stage

### 原则 5：治理不散落在 Stage 中

```text
好的方式：Stage 只发射事件 → EventBus → HookRegistry → TraceHook/MemoryHook
不好的方式：Stage 内部自己调 trace.record() + 自己写 TaskMemory + 自己判断权限
```

## 10. 一句话总结

系统以 `TaskSpec → Recipe → Stage → Executor → Tool → Facade` 为主线，`EventBus` 为治理总线，通过五层架构让 RAG 降级为能力、ReAct 受约束、治理收口为平台层——最终做到**每类能力只有一个主管理层，新增扩展只需注册不需改骨架**。
